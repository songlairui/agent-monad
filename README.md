# Agent Monad

> *A monad is just a monoid in the category of endofunctors.*
>
> 单子不过就是自函子范畴上的幺半群。

**Agent Monad** 提供三个原语 skill，让人类意图与 AI Agent 之间的切换具备**组合性** —— 就像 `Promise` 让异步操作可组合一样。

## 问题

每个 AI Agent（Claude Code、Cursor、Codex、Aider……）都是一个**自函子**：把"项目状态"映射回"项目状态"。但它们各自的内部表示不同（`CLAUDE.md` / `.cursorrules` / `codex.md`），导致：

- 切换 Agent 时上下文丢失
- 多 Agent 并行后无法合并
- 人类意图在翻译中走形

## 解法：三个 Monad 原语

| 操作 | Skill | Monad 对应 | 作用 |
|------|-------|-----------|------|
| **提升** | `agent-init` | `unit / return (η)` | 扫描项目现状，生成标准化上下文胶囊 `.agent-monad/` |
| **绑定** | `agent-bridge` | `bind / >>= ` | 从 Agent A 的遗产中提取意图+进度，转译给 Agent B |
| **合并** | `agent-merge` | `join / μ` | 多 Agent 并行工作后，合并上下文回统一状态 |

## 标准化中间表示：`.agent-monad/`

```
.agent-monad/
├── intent.md           # 人类意图（what & why，不是 how）
├── progress.jsonl      # Agent 无关的进度日志
├── conventions.md      # 从代码推断的项目约定
└── handoff.md          # 当前交接状态摘要
```

这个目录是 **monadic context** —— 包装器不是值本身，而是"意图 + 进度 + 约定"的标准化表示。

## Monad Laws 的工程含义

```
-- 左单位元：init 后立刻 bridge 到同一 Agent = 什么都没发生
unit a >>= f  ≡  f a

-- 右单位元：bridge 回自己 = 什么都没发生
m >>= unit  ≡  m

-- 结合律：A→B→C 和 (A→B)→C 结果相同，切换顺序无损
(m >>= f) >>= g  ≡  m >>= (λx → f x >>= g)
```

工程检验标准：**切换是否无损？顺序是否无关？**

## Harness Skills

除了三个 Monad 原语，还提供 **harness 工作流** skills，让 AI 在行动前丰富预见性，以置信度驱动交互频率，避免逐步 human-in-the-loop：

| Skill | 作用 |
|-------|------|
| `intent-anchor` | 意图溯源仪式，建立 session 契约 |
| `harness-graph` | 从任意种子出发，累积全景图谱 |
| `reading-breath` | 优化中文文本的节奏与密度 |
| `sancai-zhen` | **三才阵** — 统合 intent-anchor 与 harness-graph 的工作流框架 |

### 三才阵 · Usage Demo

三才阵将 harness 工作流组织为**天（全景）· 地（执行）· 人（例外）**三个阵位，以**共识页**机制取代固定仪式，AI 内部路由、自主执行。

#### 场景 1：清晰指令 → 直接执行

```
用户: 帮我把通知模块从直接调用改成事件总线

AI 内部: 路由 → 地位，生成共识页
┌──────────────────────────────────────────────┐
│ 📄 共识页  [■■■■■■■■□□] 82%  |  页幅: 中     │
│                                              │
│   ✅ 意图  — 重构通知模块，直接调用→事件总线    │
│   ✅ 终态  — 所有通知经事件总线派发，无残留      │
│   ⚠️ 现状  — 假设：存在直接调用点（待扫描）     │
│   ⚠️ 边界  — 假设：不改 UI 展示层              │
│   ❌ 风险  — 未评估                            │
│                                              │
│   82% — 有需要调整的吗？没有的话我直接开始。     │
└──────────────────────────────────────────────┘

用户: 没问题，开始

AI: 进入自主执行，中途只输出进度 ↓
┌──────────────────────────────────────────────┐
│ 📍 进度  [■■■□□] 3/5                         │
│   ✅ 扫描现有调用点 — 发现 7 处                │
│   ✅ 创建事件总线模块                          │
│   🔄 迁移调用点 (3/7)                         │
│   ○ 更新测试                                  │
│   ○ 清理旧代码                                │
└──────────────────────────────────────────────┘
```

#### 场景 2：模糊起点 → 置信度驱动追问

```
用户: 通知那块感觉不太对

AI 内部: 路由 → 地位，置信度低
┌──────────────────────────────────────────────┐
│ 📄 共识页  [■■□□□□□□□□] 22%  |  页幅: 待定    │
│                                              │

│   ❌ 终态  — 未知                              │

│                                              │
│   → 你说的"不太对"，是行为上的 bug，            │
│     还是代码结构上的不舒服？                     │
└──────────────────────────────────────────────┘

用户: 结构上的，通知发送散落在各处，没有统一入口

AI: 置信度跃升 → 重新评估
┌──────────────────────────────────────────────┐
│ 📄 共识页  [■■■■■■■□□□] 68%  |  页幅: 中      │
│                                              │
│   ✅ 意图  — 通知发送缺乏统一入口，需要收拢     │
```

每轮追问补全共识页，页面也可能生长（发现更大范围）。置信度 ≥ 85% 后自动进入执行。

#### 场景 3：执行中遇到阻碍 → 人位自动介入

```
AI 执行中发现：迁移第 4 个调用点时，该模块依赖一个未导出的内部方法

AI 内部: 地位 → 触发人/缺失
┌──────────────────────────────────────────────┐
│ 🔍 缺失                                      │
│   动作: 迁移 UserService 中的通知调用           │
│   缺失: NotificationHelper.formatPayload      │
│         是 private 方法，事件总线无法访问       │
│   类型: missing_resource                       │
│                                              │
│   建议: 将 formatPayload 提取为独立工具函数，   │
│         事件总线和原模块共用。                   │
│         这在共识页边界内，我直接处理。            │
└──────────────────────────────────────────────┘

AI: 解决后自动回到地位继续执行，不需要用户操作
```

```
│    [thought]  优先级队列可区分紧急/普通通知      │
```

归属明确时 AI 直接保存，不等"确认"。

## 安装

将 `skills/` 下的目录复制到你的 `~/.claude/skills/` 或项目的 `.claude/skills/` 中。
