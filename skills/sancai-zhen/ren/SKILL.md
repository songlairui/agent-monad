# 人 · 例外与进化

> 人者，感知之源。不在常轨中出现，只在例外时响应。

---

## 触发条件（AI 内部判断）

| 触发 | 执行 |
|------|------|
| 地位执行中发现缺失 | → [缺失检测](#缺失检测) |
| 用户提供反馈或新认知 | → [回路分发](#回路分发) |
| 图谱节点 > 50 / 阵法使用异常 | → [进化裁剪](#进化裁剪) |

---

## 缺失检测

接收地位的缺失信号，**直接给出分析和建议**，不问"确认分析？"。用户可以接受或纠正。

### 缺失分类

| 类型 | 处理 |
|------|------|
| `missing_info` | 向用户追问具体信息 |
| `missing_resource` | 补充 action 到天位，告知用户需要什么 |
| `missing_clarity` | 回到地位，补全共识页缺失维度 |
| `scope_creep` | 展示偏移，问是否扩展页幅 |
| `external_block` | 记录 blocked，建议用户先做其他事 |

### 展示格式

```
🔍 缺失
  动作: {current_step}
  缺失: {missing_condition}
  类型: {type}

  建议: {action}
```

缺失解决后，自动回到地位继续执行，不需要用户手动"回到"任何地方。

## 回路分发

收到反馈后，AI **自动判断归属并分发**，不问"确认分发？"：

| 反馈性质 | 分发到 | AI 行为 |
|---------|--------|--------|
| 新想法/认知 | 天/播种 | 自动执行 |
| 修正现有节点 | 天/更新 | 自动执行 |
| 修正当前终态 | 地/更新共识页 | 展示变更后的页面 |
| 新阻碍 | 人/缺失检测 | 见上 |
| 阵法改进建议 | 人/进化裁剪 | 见下 |

分发是 AI 的内部决策。如果分发错了，用户的后续行为自然构成纠正信号——纠正被记入阵法日志。

## 进化裁剪

### 触发条件（任一）

- 天位节点数 > 50
- 地位完成 3 次落点后
- 用户主动触发
- 阵法日志中 user_correction 率 > 30%

### 层 1 — 图谱裁剪

委托天/压缩执行。

### 层 2 — 路程裁剪

- 归档 > 30 天已完成 session → 保留摘要，删除详细步骤
- 未完成 > 14 天无更新 → 标记 `abandoned`，提示用户确认

### 层 3 — 阵法自检

分析 `~/.harness/zhen-log.json`，检测：

- 某条路由路径总是被用户纠正 → 调整路由判断
- 某个规则从未触发 → 建议移除
- 置信度评估与用户实际满意度的偏差 → 校准阈值
- 用户在高置信度下很少纠正 → 可以降低阈值，进一步减少交互

```
⚙️ 进化建议
  发现: {pattern}
  建议: {adjustment}

  接受 / 忽略 / 讨论？
```

**进化建议是唯一需要用户确认的人位操作**——因为它修改的是阵法规则本身。

## 阵法日志

每次阵法动作记录到 `~/.harness/zhen-log.json`：

```json
{
  "log_id": "log_YYYYMMDD_NNN",
  "timestamp": "",
  "position": "人",
  "sub_skill": "缺失检测",
  "trigger": "地位信号",
  "input_summary": "",
  "output": "",
  "user_correction": false,
  "confidence_at_entry": 0.82
}
```

`confidence_at_entry` 用于进化裁剪校准置信度评估的准确性。

## 人位守则

1. **被动响应**：不主动打断执行，只在被触发时介入
2. **自动分发**：回路不问用户确认，分发是内部决策
3. **纠正即信号**：用户纠正记入日志，驱动阵法进化
4. **阻碍即图谱缺口**：每个阻碍最终变成天位图谱中的新节点
5. **进化建议需确认**：唯一的例外——修改规则本身需要用户同意
